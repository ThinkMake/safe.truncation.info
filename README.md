# Safe Truncation Information

A Jekyll site cataloguing words that produce offensive or misleading results when truncated — for example, "Analgesic" truncated to "Anal...". The project helps developers and content designers identify and handle these cases safely.

Live at [safe.truncation.info](https://safe.truncation.info).

Site conceived, designed and built by [Mark Hurrell](https://mhurrell.co.uk) with Claude Code.

---

## How content is generated

Content is defined in language-specific YAML data files in `_data/`. The English list is at `_data/en.yml`. Each entry has the following fields:

```yaml
- word: Analgesic
  bad_word: anal
  problem: "<strong>Anal</strong>gesic"
  definition: "Analgesic refers to any substance that relieves pain..."
  bad_word_definition: "Anal is an adjective derived from the Latin anus..."
  lang: en
```

| Field | Purpose |
|---|---|
| `word` | The full, safe word |
| `bad_word` | The offensive string produced by truncation |
| `problem` | HTML highlighting which part of the word is problematic |
| `definition` | Definition of the full word |
| `bad_word_definition` | Definition of the bad word, explaining its range of meanings |
| `lang` | BCP 47 language tag — used as the `lang` attribute on each `<section>` element |

The site layout (`_layouts/en.html`) iterates over `site.data.en`, sorted alphabetically, to render each word as a `<section>` with a `<details>` element. The nav panel mirrors the same list.

To add or edit a word, edit `_data/en.yml` directly. No other files need to change.

## API

Each language exposes its word list as a static JSON file at `/<lang>.json` — for example:

```
https://safe.truncation.info/en.json
```

The JSON is generated by Jekyll from the same data file. Each entry follows this shape:

```json
{
  "word": "Analgesic",
  "bad_word": "anal",
  "definition": "Analgesic refers to any substance...",
  "bad_word_definition": "Anal is an adjective derived from the Latin anus..."
}
```

### Suggested usage

1. Fetch `en.json` and cache the list.
2. When truncating a string, check whether any `word` in the list matches (case-insensitive).
3. If there is a match, verify that your truncated output does not begin with the corresponding `bad_word`.

## Adding a new language

The project is structured to support multiple languages. Each language is a self-contained unit:

1. **Add a data file** — create `_data/<lang>.yml` using the same field structure as `en.yml`. Set `lang` on every entry to the appropriate BCP 47 tag (e.g. `fr`, `de`).

2. **Add a layout** — create `_layouts/<lang>.html`. Copy `_layouts/en.html` and update the `site.data` reference from `en` to the new lang code, and pass the matching `lang` value to the footer include:
   ```liquid
   {% assign sorted_items = site.data.fr | sort_natural: "word" %}
   {% include footer.html lang="fr" %}
   ```

3. **Add a JSON endpoint** — create `<lang>.json` at the project root. Copy `en.json` and update the `site.data` reference to match.

4. **Add an index page** — create a page (e.g. `fr/index.md`) pointing to the new layout:
   ```yaml
   ---
   layout: fr
   ---
   ```

The `lang` attribute on each `<section>` element is already wired up from the data file, so screen readers and language tools will correctly identify the language of each word entry without affecting the English site navigation.

### Note on `<html lang>`

The `<html lang="en">` attribute is intentionally fixed to English across all language variants. Site navigation, the header, footer, and UI chrome are all in English. Only the word entries themselves are in the target language — these are scoped with the per-section `lang` attribute rather than a page-level override.
